## Overview

This spec covers five enhancements to the Medusa plugin that bring it to parity with (and beyond) the Saleor App roadmap. The theme is **DX and merchant self-serve** — turning developer tasks into dashboard actions and giving merchants visibility into the agent commerce channel.

All work is within `packages/core/src/`.

#### Data Flow

Settings are stored via Medusa's **Store metadata** (not a custom table). This keeps things simple and avoids migrations.

```typescript
// Read settings
const store = await sdk.client.fetch<StoreResponse>("/admin/store")
const settings = store.metadata?.agentic_commerce ?? defaults

// Write settings
await sdk.client.fetch("/admin/store", {
  method: "POST",
  body: { metadata: { agentic_commerce: settings } }
})
```

**Settings schema stored in `store.metadata.agentic_commerce`:**

```typescript
type AgenticCommerceSettings = {
  ucp_enabled: boolean        // default: true
  acp_enabled: boolean        // default: true
  store_name: string          // default: from module options
  store_description: string
  storefront_url: string
  api_key: string             // ACP bearer token
  signature_key: string       // HMAC secret
}
```

#### Service Changes

The `AgenticCommerceService` needs a method to reload settings at runtime without restart:

```typescript
// Add to AgenticCommerceService
async refreshSettings(query: any): Promise<void> {
  const { data: [store] } = await query.graph({
    entity: "store",
    fields: ["metadata"],
  })
  const settings = store?.metadata?.agentic_commerce
  if (settings) {
    if (settings.store_name) this.storeName = settings.store_name
    if (settings.store_description) this.storeDescription = settings.store_description
    if (settings.storefront_url) this.storefrontUrl = settings.storefront_url
    if (settings.api_key) this.apiKey = settings.api_key
    if (settings.signature_key) this.signatureKey = settings.signature_key
  }
}
```

The middleware should call `refreshSettings()` periodically (not on every request — cache for 60 seconds).

#### Protocol Toggle

When UCP or ACP is disabled via settings, the middleware should return `404` for those route groups. The `.well-known` endpoints should still work but reflect the disabled state.

#### Quick Test Buttons

The "Test UCP Discovery" and "Test ACP Discovery" buttons make a fetch to the backend's own `.well-known/ucp` and `.well-known/acp.json` endpoints from the browser and display the response status + payment handler count. This gives merchants instant feedback that the integration is working.

#### Implementation Notes

- Use `defineRouteConfig({ label: "Agentic Commerce" })` for sidebar
- Use `@medusajs/ui` components: `Container`, `Heading`, `Input`, `Button`, `Badge`, `Switch`, `Label`, `Copy`
- The page is read/write — loads current settings on mount, saves on button click
- API key fields should be masked by default with a "show" toggle
- "Regenerate" buttons generate a crypto-random key client-side and save it

## 3. Order Lifecycle Subscribers

**Goal:** Push real-time order updates to agents via webhook, populating the currently-empty `fulfillment.events[]` in protocol responses.

### Current State

Only one subscriber exists: `order-placed-webhook.ts` listening to `order.placed`.

### New Subscribers to Add

All follow the same pattern as the existing subscriber — resolve the order, find the cart's `agent_webhook_url`, send webhook if present.

#### 3a. `order-updated-webhook.ts`

```typescript
// src/subscribers/order-updated-webhook.ts
export const config: SubscriberConfig = {
  event: "order.updated",
}
```

**Webhook event type:** `order_update`  
**Payload:** Same structure as `order_create` but with current status, updated totals.

**Deduplication:** The `order.updated` event can fire frequently. Add a debounce check — store `last_webhook_sent_at` in order metadata and skip if < 5 seconds since last send.

#### 3b. `fulfillment-created-webhook.ts`

```typescript
// src/subscribers/fulfillment-created-webhook.ts
export const config: SubscriberConfig = {
  event: "fulfillment.created",
}
```

**Webhook event type:** `fulfillment_create`  
**Payload:**

```json
{
  "type": "fulfillment",
  "checkout_session_id": "cart_01JF...",
  "order_id": "order_01JF...",
  "fulfillment_id": "ful_01JF...",
  "status": "shipped",
  "tracking_number": "1Z999AA10123456784",
  "carrier": "UPS",
  "items": [
    { "product_id": "prod_01...", "variant_id": "variant_01...", "quantity": 1 }
  ]

```

**Note:** Fulfillment events need to resolve the order from the fulfillment, then find the cart. The chain is: `fulfillment → order → cart → metadata.agent_webhook_url`.

### Retry Logic

Add basic retry to `sendWebhookEvent()` in `AgenticCommerceService`:

```typescript
async sendWebhookEvent(params: {
  url: string
  event_type: string
  payload: Record<string, unknown>
  maxRetries?: number
}): Promise<{ status: number; success: boolean }> {
  const maxRetries = params.maxRetries ?? 3
  const delays = [0, 2000, 10000] // immediate, 2s, 10s

  for (let attempt = 0; attempt <= maxRetries; attempt++) {
    if (attempt > 0) await sleep(delays[attempt])
    try {
      const response = await fetch(params.url, { ... })
      if (response.ok) return { status: response.status, success: true }
      if (response.status >= 400 && response.status < 500) {
        // Client error — don't retry
        return { status: response.status, success: false }
      }
    } catch { /* network error — retry */ }
  }
  return { status: 0, success: false }
}
```

## Technical Notes

### Medusa v2 Admin Extension Conventions

- Widget files go in `src/admin/widgets/` — auto-discovered by the plugin build
- Route files go in `src/admin/routes/` — auto-discovered
- Components must be **arrow functions** (not class components)
- Use `@medusajs/ui` for all UI components: `Container`, `Heading`, `Input`, `Button`, `Badge`, `Switch`, `Text`, `Table`, `Copy`
- Use `@medusajs/icons` for icons
- Use `defineWidgetConfig({ zone: "..." })` and `defineRouteConfig({ label: "..." })`
- Use `sdk.client.fetch()` for API calls from admin pages (auto-handles auth)

### Key Injection Zones

| Zone | Where it appears |
|------|-----------------|
| `order.details.before` | Top of order detail page |
| `order.details.after` | Bottom of order detail page |
| `order.list.before` | Top of order list page |
| `product.details.before` | Top of product detail page |

Verify `home.before` exists in the current Medusa version — if not, use `order.list.before` for the analytics widget.

### Store Metadata Pattern

Using `store.metadata` for settings avoids:
- Custom database tables
- Custom migrations
- Extra module registration

Tradeoff: metadata is a JSON blob, so no database-level validation. The settings page handles validation client-side + the service validates on load.

### Testing

- Admin UI: Manual testing in the browser (Medusa admin extensions don't have a standard test harness)
- Subscribers: Write unit tests that mock the container and verify webhook calls
- Analytics endpoint: Integration test via `GET /admin/agentic-commerce/analytics`

## Open Questions

1. **Settings storage:** Store metadata vs a custom module with its own table? Metadata is simpler but has no migration path if the schema evolves. Recommendation: start with metadata, migrate to a dedicated table in v0.3 if needed.

2. **UCP/ACP toggle behavior:** When a protocol is disabled, should we return 404 or a specific error? Recommendation: 404 — keeps it clean for agents that probe capabilities.

3. **Analytics performance:** Querying all orders and checking cart metadata could be slow at scale. Consider a denormalized approach: maintain a counter in store metadata that subscribers increment. Good enough for v0.2, optimize later.

4. **Available Medusa v2 events:** Verify the exact event names for fulfillment tracking updates. The event naming convention in Medusa v2 may differ from what's listed above. Check `@medusajs/utils` event constants.
